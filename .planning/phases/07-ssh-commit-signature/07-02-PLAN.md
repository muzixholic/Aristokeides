# Plan 07-02: SSH Commit Signature - Git Integration & Push Hooks Layer

이 계획은 SSH Push와 HTTP Push가 성공적으로 완료되는 시점을 캐치하여, 푸시로 인해 새롭게 유입된 커밋들의 해시 목록을 LibGit2Sharp로 추적하고 각 커밋의 SSH 디지털 서명을 자동으로 검증하여 DB에 적재하는 백엔드 연동 파이프라인을 구축합니다.

---

## Tasks

### 1. 통합 서명 검증 오케스트레이션 서비스 구현
<read_first>
- Aristokeides.Api/Data/AppDbContext.cs
- Aristokeides.Api/Services/Ssh/SshSignatureVerifier.cs
- Aristokeides.Api/Services/Ssh/SshSignatureParser.cs
</read_first>
<action>
1. `Aristokeides.Api/Services/Ssh/SshSignatureVerificationService.cs` 클래스를 생성합니다.
2. 이 서비스는 다음 주요 기능을 비동기로 수행합니다:
   - `VerifyNewCommitsAsync(string repoPhysicalPath, string oldOid, string newOid, Guid repositoryId)`:
     - Push로 인해 추가된 커밋 범위(`oldOid..newOid`)를 LibGit2Sharp `CommitFilter` 또는 `repo.Commits.QueryBy`를 활용해 조회합니다. 만약 `oldOid`가 `0000000000000000000000000000000000000000` (신규 브랜치/태그 생성) 이라면, `newOid`로부터 조상 방향으로 탐색하되 기존에 이미 서명 검증이 완료된 커밋을 만나거나 루트 커밋을 만날 때까지 추적합니다.
     - 각 추적된 커밋 객체에 대해 `ObjectDatabase.RetrieveObjectRaw(commitOid)` 또는 `ExtractSignature`를 호출해 SSH 서명 텍스트와 서명 페이로드 바이트 배열을 추출합니다.
     - 서명이 발견되지 않을 경우: Status = `NoSignature`로 DB에 저장합니다.
     - 서명이 발견될 경우:
       - `SshSignatureParser`를 사용해 서명 데이터에서 공개키 바이너리를 추출하고 SHA-256 지문을 구합니다.
       - DB의 `SshKeys` 테이블에서 해당 지문과 매칭되는 등록 키를 조회합니다.
       - 일치하는 키가 있을 경우: 해당 공개키 원본으로 `SshSignatureVerifier.Verify(...)`를 호출해 검증합니다.
         - 검증 성공 시: Status = `Verified`, `SignerUserId` = 해당 키의 소유자 User ID
         - 검증 실패 시: Status = `Invalid`, `SignerUserId` = null
       - 일치하는 키가 없을 경우 (Unknown):
         - 추출된 공개키 데이터 자체를 기반으로 가상 공개키를 복원하여 `SshSignatureVerifier`로 서명의 무결성 자체를 검증합니다.
         - 무결성 검증 성공 시: Status = `Unknown` (신뢰성은 확인되었으나 등록되지 않은 키)
         - 무결성 검증 실패 시: Status = `Invalid`
     - 검증된 결과를 `CommitSignature` 엔티티 객체로 생성하고 DB에 `Upsert` 처리합니다. (동일 커밋에 대한 중복 검증 시 덮어쓰기 지원)
</action>
<acceptance_criteria>
- `SshSignatureVerificationService`가 구현되어 컴파일이 성공함.
- 신규 브랜치 생성(`oldOid`가 0인 상황)과 일반 푸시 업데이트 상황 모두에 대응하여 LibGit2Sharp로 대상 커밋 목록을 정확하게 열거할 수 있음.
</acceptance_criteria>

### 2. SSH Push 연동 훅 통합
<read_first>
- Aristokeides.Api/Services/Ssh/SshCommandBridge.cs
- Aristokeides.Api/Services/Ssh/SshSignatureVerificationService.cs
</read_first>
<action>
1. `SshCommandBridge.cs` 생성자에 `IServiceProvider` 혹은 `SshSignatureVerificationService` 의존성을 주입합니다.
2. `RunGitCommandAsync`에서 `commandName`이 `"git-receive-pack"`일 때, Git 프로세스가 종료(`process.WaitForExitAsync()` 완료)되고 `process.ExitCode == 0`인 경우:
   - Git 저장소의 레퍼런스 변경 내역을 수집합니다. (프로세스 실행 전후로 target ref의 OID 정보를 읽어 차이를 추출하거나, `git-receive-pack` 실행 로그/결과로부터 업데이트 정보를 확인합니다. 단순하고 안전한 방식은, Push 프로세스가 실행되기 전 브랜치의 Head OID와 실행 후 브랜치의 Head OID를 백업/비교하여 변경 범위를 알아내는 것입니다.)
   - 알아낸 변경 범위(`oldOid`, `newOid`)를 인자로 전달하여 `_signatureVerificationService.VerifyNewCommitsAsync(...)`를 백그라운드로 안전하게 구동합니다.
</action>
<acceptance_criteria>
- SSH 프로토콜을 통해 `git push`가 성공적으로 끝났을 때, 새 커밋들이 `CommitSignatures` 테이블에 올바른 상태값과 함께 자동 기록됨.
</acceptance_criteria>

## Verification Criteria

1. `PushHookIntegrationTests`
   - 가상의 Git Push 동작을 트리거(레퍼런스 OID 수동 업데이트 모사)하여 `SshSignatureVerificationService`가 새 커밋들을 스캔하고 올바르게 DB에 인서트하는지 모의 테스트 검증.
   - `Verified`, `Unknown`, `NoSignature` 각 서명 패턴별 레코드가 테이블에 매칭 삽입되는지 확인.

### Manual Check
- 로컬 Git 클라이언트를 사용해 SSH/HTTP 각각으로 SSH 서명 커밋을 Push한 뒤, `CommitSignatures` 테이블에 신규 행이 추가되었는지 쿼리를 실행해 직접 검증.
