# Plan 07-01: SSH Commit Signature - Database & Crypto Layer

이 계획은 SSH 커밋 서명 검증 결과를 보존할 데이터베이스 엔티티와 마이그레이션을 구성하고, SSH 서명 봉투(SSHSIG)로부터 공개키를 파싱하며, `ssh-keygen` 도구를 안전하게 호출하여 서명을 해독하는 백엔드 코어 모듈을 구축합니다.

---

## Tasks

### 1. CommitSignature 데이터 모델 및 저장소 정책 추가
<read_first>
- `Aristokeides.Api/Models/Repository.cs`
- `Aristokeides.Api/Models/SshKey.cs`
- `Aristokeides.Api/Data/AppDbContext.cs`
</read_first>
<action>
1. `Aristokeides.Api/Models/CommitSignature.cs` 파일을 생성하고 다음 속성을 포함한 엔티티를 작성합니다:
   - `Id` (int, PK)
   - `CommitHash` (string, MaxLength 50)
   - `RepositoryId` (Guid, FK to Repository)
   - `SignerUserId` (int?, FK to User, Nullable, OnDelete: SetNull)
   - `Status` (string, Verified / Invalid / Unknown / NoSignature)
   - `Algorithm` (string?, ssh-ed25519 / ssh-rsa / ecdsa-sha2-nistp256 등)
   - `KeyFingerprint` (string?, SSH 공개키 지문)
   - `VerifiedAt` (DateTime)
2. `Aristokeides.Api/Models/Repository.cs`에 `RequireSignedCommits` (bool, 기본값 false) 프로퍼티를 추가합니다.
3. `Aristokeides.Api/Data/AppDbContext.cs`에 `DbSet<CommitSignature>`를 추가하고, `OnModelCreating`에서 다음 제약조건을 설정합니다:
   - `RepositoryId + CommitHash` 복합 유니크 인덱스 생성.
   - `CommitSignature.SignerUserId` 관계를 `DeleteBehavior.SetNull`로 매핑.
</action>
<acceptance_criteria>
- `AppDbContext.cs`에 `CommitSignatures`가 올바르게 정의됨.
- `Repository.cs`에 `RequireSignedCommits` 필드가 존재함.
- `CommitSignature` 엔티티 정의가 컴파일 성공함.
</acceptance_criteria>

### 3. OpenSSH 서명 바이너리 파서 (`SshSignatureParser`) 구현
<read_first>
- `Aristokeides.Api/Services/Ssh/SshKeyParser.cs` (이진 스트림 읽기 헬퍼 재사용 가능성 확인)
</read_first>
<action>
1. `Aristokeides.Api/Services/Ssh/SshSignatureParser.cs` 파일을 생성하고 SSHSIG 바이너리 디코더를 작성합니다:
   - `-----BEGIN SSH SIGNATURE-----`와 `-----END SSH SIGNATURE-----` 사이의 Base64 문자열을 디코딩합니다.
   - 디코딩된 바이트 배열에서 Magic String `"SSHSIG"` (6바이트) 및 Version `1` (4바이트)을 검증합니다.
   - 공개키 길이 `L` (uint32, 빅엔디안)을 읽고, 그 뒤 `L` 바이트만큼 공개키 원시 바이트를 추출합니다.
   - 추출한 공개키 원시 바이트로부터 공개키의 알고리즘(예: `ssh-ed25519`)을 추출하여 파싱합니다.
   - 공개키 원시 바이트 전체의 SHA-256 해시를 계산하여 `"SHA256:[Base64]"` 지문 포맷을 빌드하는 헬퍼 메서드를 정의합니다.
</action>
<acceptance_criteria>
- 유효한 SSH 서명 텍스트 입력 시 공개키 이진 데이터, 알고리즘 유형, 그리고 정확한 SHA-256 Fingerprint가 반환됨.
- 잘못된 SSHSIG 매직 헤더나 버전을 가진 경우 `ArgumentException`이 안전하게 발생함.
</acceptance_criteria>

### 4. `ssh-keygen` 연동 서명 검증기 (`SshSignatureVerifier`) 구현
<read_first>
- `Aristokeides.Api/Services/Ssh/SshSignatureParser.cs`
- `Aristokeides.Api/Data/AppDbContext.cs`
</read_first>
<action>
1. `Aristokeides.Api/Services/Ssh/SshSignatureVerifier.cs` 파일을 생성하고 `ssh-keygen -Y verify` 프로세스를 호출하는 검증 함수를 설계합니다.
   - 검증 대상 커밋 페이로드 바이트 배열과 SSH 서명 텍스트, 그리고 타겟 공개키 문자열을 인자로 받습니다.
   - 임시 폴더 내에 고유 식별자(Guid)를 파일명으로 가진 3개의 임시 파일(`allowed_signers.tmp`, `payload.tmp`, `sig.tmp`)을 생성합니다.
   - `allowed_signers.tmp`에 `principal ssh-ed25519 [PublicKeyBase64]` 형식의 줄을 기록합니다.
   - `payload.tmp`에 페이로드 바이트를 쓰고, `sig.tmp`에 서명 텍스트를 기록합니다.
   - `ProcessStartInfo`를 사용해 다음 OS 명령을 실행합니다:
     `ssh-keygen -Y verify -f [allowed_signers_path] -I principal -n git -s [sig_path] < [payload_path]`
     (`UseShellExecute = false`, `RedirectStandardInput = true`를 적용하여 payload 데이터를 StandardInput 스트림으로 직접 라이팅할 수도 있습니다.)
   - 프로세스 종료 코드(ExitCode)가 `0`이면 검증 성공(`true`), 그렇지 않으면 실패(`false`)로 판정합니다.
   - 검증 완료 후, 생성했던 모든 임시 파일들을 `try-finally` 블록을 활용해 완전히 파괴/삭제하여 찌꺼기를 남기지 않습니다.
</action>
<acceptance_criteria>
- 유효한 서명과 페이로드로 검증 요청 시 `true`가 반환됨.
- 변조된 페이로드 혹은 불일치 서명 요청 시 `false`가 반환됨.
- 검증 동작 완료 후 임시 폴더에 어떠한 파일 찌꺼기도 잔존하지 않음.
</acceptance_criteria>
