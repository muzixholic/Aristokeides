# Phase 21: 웹훅 및 외부 연동 - User Acceptance Testing (UAT)

**Date:** 2026-06-10
**Status:** Completed

## 1. Acceptance Test Cases

### UAT-1: Git Push 이벤트를 통한 비동기 웹훅 발송 검증
- **테스트 환경:** 웹훅 수신 테스트 서버(예: Webhook.site 등) 및 Aristokeides 로컬 개발 서버
- **시나리오:**
  1. 저장소 설정 페이지로 이동하여 신규 웹훅을 등록합니다. (타입: `Generic`, URL: 테스트 수신 서버 URL, 이벤트: `push`)
  2. 로컬 저장소에서 새로운 커밋을 작성하고 서버로 `git push`합니다.
- **예상 결과:**
  - 푸시 완료 후, 수신 테스트 서버에 실시간으로 JSON 페이로드가 수신됩니다.
  - 헤더에 `X-Aristokeides-Event: push` 및 고유 UUID가 전송됩니다.
  - 전송 로그 테이블(`WebhookDeliveries`)에 성공(200) 이력이 안전하게 영속화됩니다.

### UAT-2: HMAC-SHA256 서명 보안 검증
- **시나리오:**
  1. 저장소에 웹훅을 수정/등록합니다. 이때 Secret Key로 `my_secret_key`를 설정합니다.
  2. 이슈(Issue)를 새로 개설하여 이벤트를 트리거합니다.
  3. 수신 서버에 도달한 요청 헤더 `X-Aristokeides-Signature-256` 값을 확인합니다.
  4. 받은 페이로드 바디 전체를 `my_secret_key` 및 HMAC-SHA256 알고리즘을 사용해 해시 계산하고 전송된 헤더 서명과 일치하는지 비교합니다.
- **예상 결과:**
  - 서명 헤더 `X-Aristokeides-Signature-256`이 정상 동반됩니다.
  - 수동으로 계산한 해시값과 헤더 서명값이 정확히 일치하여 위조 방지가 정상 검증됩니다.

### UAT-3: 웹 UI 이력 조회 및 수동 재전송(Redelivery) 검증
- **시나리오:**
  1. 일부러 수신할 수 없는 잘못된 URL(예: `https://invalid-host-name-xyz.com`)로 웹훅을 임시 등록합니다.
  2. 이슈를 생성해 이벤트를 강제 발동시킨 뒤, 저장소 웹훅 설정 화면에서 전송 이력 목록을 봅니다.
  3. 실패한 기록의 상세 보기 버튼을 클릭하여 요청과 에러 로그를 확인합니다.
  4. 웹훅 대상 URL을 정상적인 테스트 수신 URL로 변경합니다.
  5. 앞서 실패했던 전송 이력 우측의 "재전송" 버튼을 클릭합니다.
- **예상 결과:**
  - 최초 실패 전송은 이력 목록에 빨간색 실패 아이콘 및 상세 네트워크 오류 로그가 노출됩니다.
  - "재전송" 단추 클릭 즉시, 수신 서버로 동일한 페이로드가 재전송되어 도달합니다.
  - 재전송된 요청 역시 성공 이력으로 리스트에 동적으로 업데이트됩니다.

### UAT-4: Slack 및 Discord 형식 템플릿 변환 검증
- **시나리오:**
  1. 웹훅을 추가하되 타입을 `Slack`으로 설정하고 슬랙 채널 Incoming Webhook URL을 기입합니다.
  2. 또 다른 웹훅을 추가하고 타입을 `Discord`로 설정하여 디스코드 웹훅 URL을 등록합니다.
  3. 새로운 이슈를 작성합니다.
- **예상 결과:**
  - 슬랙 채널에 `✏️ *[repo]* Issue #X opened...` 형식의 서식 가공된 슬랙 메시지가 노출됩니다.
  - 디스코드 채널에 임베드 서식이나 마크다운이 적용된 디스코드 전용 레이아웃 메시지가 정상 수신됩니다.
