# Phase 20: Git LFS (Large File Storage) 지원 - Security Threat Model

**Date:** 2026-06-10
**Status:** Completed

## 1. Threat Analysis & Mitigations

### 1.1. 파일 시스템 탐색 공격 (Directory Traversal)
- **위협:** 악의적인 사용자가 OID 값에 `../` 등을 섞어 LFS 업로드/다운로드 경로를 오염시키고 시스템 내의 임의의 경로에 있는 파일을 탈취하거나 덮어쓰려 할 수 있습니다.
- **대응책:**
  - OID 값에 대한 엄격한 형식 검증을 수행합니다. OID는 반드시 64자리의 16진수 문자열(SHA-256) 형태여야만 하므로, 이를 정규표현식(`^[a-fA-F0-9]{64}$`)으로 유효성 체크를 진행하여 그 외의 값은 즉시 차단합니다.
  - 파일 저장 및 조회 시 지정된 디렉토리 외부 경로에 액세스하지 못하도록 물리적 절대 경로를 확인하고 비교합니다.

### 1.2. 시한성 토큰 탈취 및 오용
- **위협:** LFS Batch API의 2차 전송용 임시 토큰이 영구적이거나 오래 유효할 경우, 비인가자가 이 토큰을 획득하여 중요 저장소의 대용량 파일을 무단으로 내려받거나 악성 파일을 업로드할 수 있습니다.
- **대응책:**
  - 임시 토큰의 수명을 기본 1시간 이내로 한정하여 만료 시간이 지난 토큰의 재사용을 원천적으로 제한합니다.
  - JWT 토큰 내부 클레임에 `RepositoryId`, `Oid`, `Action` ("upload" 또는 "download")을 포함시키고, 전송 API 측에서 해당 클레임이 실제 요청하려는 매개변수와 완벽하게 일치하는지 엄밀히 대조하여 교차 검증합니다.

### 1.3. 무제한 파일 업로드를 통한 디스크 고갈 공격 (DoS)
- **위협:** 인가된 사용자가 비정상적으로 큰 크기의 대용량 파일을 계속 업로드하여 LFS 로컬 스토리지의 디스크 공간을 소진시킬 수 있습니다.
- **대응책:**
  - Batch API 요청 시 개별 파일 크기에 대한 서버 측 최대 제한 임계값(예: 저장소 요금제 또는 시스템 설정에 따라 2GB 등)을 정의하고 초과 시 처리를 반려합니다.
  - PUT 업로드 시 스트림을 처리하며 지정된 `size`를 넘어서서 데이터가 전송되는 경우 커넥션을 즉시 강제 종료(Abrupt Connection Close)하여 스토리지 오버런을 방지합니다.

### 1.4. Locks API 권한 도용 및 잠금 강제 해제
- **위협:** 권한이 없는 사용자가 타인의 잠금을 임의로 해제(Unlock)하여 소스 코드의 소유권을 강탈하거나 쓰기 충돌을 야기할 수 있습니다.
- **대응책:**
  - `Locks API` 엔드포인트에 Basic Auth 인증을 필수로 요구합니다.
  - 락 해제(`POST /unlock`) 시 해당 사용자가 락의 원소유자(Owner)이거나 저장소의 `Admin` 권한을 가진 경우에만 허용하고, 일반 `Write` 권한 사용자의 타인 락 해제 요청은 403 Forbidden으로 엄격히 거부합니다.
