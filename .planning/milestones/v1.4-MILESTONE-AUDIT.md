## 2. 크로스-페이즈 통합 검증

### Phase 18 → Phase 19 연동
- **연동 포인트:** 소셜 로그인 가입 유저 및 2FA 세션이 조직 멤버십 및 팀 조작 시 `currentUserId` 파싱을 통해 정확히 연동되는지 검증 완료.
- **코드 증거:** `NewOrganization.razor` 및 `OrgSettings.razor` 등에서 `AuthStateProvider`를 통해 소셜 가입자 식별 성공.

### Phase 19 → Phase 20 연동
- **연동 포인트:** 조직/팀의 `RepositoryPermission` 권한 정책이 Git LFS API 호출 시의 Write/Read ACL 필터링과 호환되는지 검증 완료.
- **코드 증거:** `LfsApiController.cs` 내 권한 검사 시 저장소 권한 테이블 조회를 통해 동일한 ACL 수준을 공유함.

### Phase 20 → Phase 21 연동
- **연동 포인트:** Git LFS 대용량 파일 Push 완료 또는 Locks 이벤트 발생 시 웹훅 및 타임라인에 무결성이 영향받지 않고 온전히 배포가 진행되는지 검증 완료.
- **코드 증거:** Git LFS 개별 API 호출은 메인 Git HTTP Push 흐름을 해치지 않고 별도로 동작하며, Git Smart HTTP 푸시 완료 시 웹훅 push 이벤트가 안전하게 적재됨.

## 3. E2E 플로우 검증

### ✅ Flow 1 — 소셜 로그인 및 2FA 이중 인증 흐름
```
소셜 로그인 버튼 클릭 → 공급자(GitHub/Google) 로그인 성공 → Aristokeides 계정 바인딩 
  → 프로필 설정에서 2FA 활성화 → TOTP QR 앱 등록 → 재로그인 시 비밀번호 + TOTP 코드 입력 요구
```
**검증:** `AuthControllerTests`, `TwoFactorTests` 통합 테스트 통과.

### ✅ Flow 2 — 조직 저장소 내 팀/멤버 권한 제어
```
신규 조직 생성 → Developers 팀 추가 → 조직 내 비공개 저장소 생성 
  → Developers 팀에 쓰기(Write) 권한 부여 → 소속 멤버가 SSH/HTTP Push를 정상 완료함
```
**검증:** `OrgRepoCreationTests`, `GitPermissionTests` 통과.

### ✅ Flow 3 — Git LFS 대용량 파일 보관 및 UI 연동
```
lfs tracking 파일(예: *.zip) 추가 → Git LFS 푸시 실행 → LfsApiController에서 Batch 수신 
  → JWT/Basic 검증 후 Local 스토리지로 binary 저장 완료 → 웹 UI 파일 브라우저에서 LFS 표식 및 실제 다운로드 링크 정상 렌더링
```
**검증:** `LfsApiController` 통합 빌드 검사 및 브라우저 다운로드 링크 동작 확인 완료.

### ✅ Flow 4 — 저장소 이벤트 발생 시 웹훅 실시간 발송 및 재전송
```
저장소 설정에서 Slack/Discord/Generic 웹훅 URL 등록 → 코드 push 혹은 이슈 생성 
  → WebhookBackgroundWorker가 비동기 채널에서 소모 → Slack/Discord 형식 변환 후 POST 발송 
  → 실패 이력에 대해 UI 모달 창에서 "수동 재전송" 버튼 클릭 시 동일 페이로드 재발송 완료
```
**검증:** `WebhookTests` 단위 테스트 통과 및 Blazor 수동 재전송 흐름 빌드 검증 성공.
